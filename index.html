Artwishlist Definitions and Workflows

Code Manifesto, Best Practices, Do's and Dont's

Definitions

Roles:

  • Stakeholders (Christian, Dimitris)
  • Product Owners (Romanos)
  • In-house Dev team (Romanos, Apostolis)
  • Devs (Apostolis, Romanos, Freelancers)

Pivotal:

  • Finished -> Done all commits
  • Delivered -> Pushed to develop and deployed on staging
  • Accepted -> All works fine, pushed to master and deployed to production

Code Manifesto (CM)

  1. Create new branch feat/story-name or fix/story-name or chore/story-name
  2. Always rebase
  3. Always include in commits [#story-id]
  4. On last commit [Finishes #story-id]
  5. Create tests (Unit, Functional, Behavioral Tests)
  6. Good naming classes/methods/variables
  7. Create comments for where it gets too complicated
  8. Run http://rubocop.readthedocs.io/en/ against all RoR file changes
  9. Run http://www.coffeelint.org/ against all Coffeescript file changes
  10. Run https://github.com/brigade/scss-lint against all SCSS file changes

Code Review (CR)

  1. Is code readable
    1. Good naming classes/methods/variables
    2. Create comments for where it gets too complicated
  2. Is OOP used?
  3. Check code security
  4. Check code performance

Security Best Practices:

  • Enable and enforce 2WA wherever possible
  • Regenerate passwords and API tokens whenever a collaborator leaves or/and every month
  • Roles and Accounts in AWS, Github, PivotalTracker, Apps. Everyone is responsible for his/her own actions

Work Flow Protocol

Stakeholders:

  1. Check for Bugs
  2. Determine new Features
  3. Create Roadmap

Stakeholders + Product Owners:

  1. Provide product owner with docs, images, general requirements
  2. Set priorities on which features are most needed
  3. Check functionality on staging
  4. Mark story as Accepted

Product Owners:

  1. Create Epics
  2. Create New Pivotal Stories
    1. Gather all information regarding the 'issue'
    2. Almost all stories should be part of an epic
    3. A new story should be as small as possible
    4. Add subtasks in each story (fat subtasks, thin description)
    5. Provide examples and images/docs whenever possible
  3. Assign stories to Devs

In-house Dev team:

  1. Calculates and add Points for stories
  2. Accept pull request (Review)
  3. Mark story as delivered
  4. Deploy on staging
  5. Check functionality on staging
  6. Push code to master
  7. Deploy on master

Devs:

  1. Communicate subtasks modification on pivotal:
    1. Change subtasks as needed
    2. Writing explanation why a change was needed as comment with reference to requester
    3. Continue working until feedback
  2. Writes code according to Code Manifesto (CM)
  3. Pull requests to develop branch
